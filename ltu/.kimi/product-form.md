# LTU - 产品形态决策

## 决策: 电视 + 摄像头 + 麦克风

### 选择理由
| 因素 | 电视方案的优势 |
|------|---------------|
| **渗透率** | 家家都有，无需额外购买硬件 |
| **屏幕** | 大屏适合视觉反馈，比手机/平板护眼 |
| **固定场景** | 有"仪式感"，家长知道孩子在干嘛 |
| **输入丰富** | 摄像头能看到表情/动作/距离，比纯语音多维 |
| **快速验证** | 可以OTT应用/小程序形式上线，周期短 |

### 核心痛点假设
> **替代陪伴优先** —— 先解决"有人陪"，再谈培养、引导、理解

家长最缺的不是教育方法，是**时间和精力**。当家长必须做饭/工作/处理事情时，孩子需要有人回应。高质量的"替代陪伴"本身就是价值。

---

## 关键设计挑战

### 1. 电视的被动性 → 主动互动
电视传统是"看"的，但陪伴需要"对话"。
- 如何让孩子愿意对着电视说话？
- 如何设计"召唤"机制（语音唤醒？手势？特定动作？）
- 内容如何穿插互动节点，而不是单向播放？

### 2. 身体活动范围
3岁孩子坐不住，会跑来跑去。
- 摄像头需要多大视角？固定机位还是可旋转？
- 是否需要"跟随"机制（孩子离开沙发，互动暂停/提醒）
- 设计多少"身体活动"内容（跳舞、模仿动作）来消耗精力？

### 3. 隐私与安全
摄像头在客厅，家长敏感。
- **物理开关**：必须有实体摄像头遮蔽/旋转
- **指示灯**：何时在录像/录音，需要明确信号
- **本地优先**：视频流不上云，AI在本地或边缘计算
- **家长授权**：每次启动需要家长端确认

### 4. 替代陪伴的边界
家长担心"孩子只听AI的，不听我的"。
- 如何设计让家长始终是"最高权限"？
- AI如何主动"邀请"家长参与（"妈妈想听你刚才唱的歌"）
- 内容是否区分"家长在场模式"和"替代陪伴模式"？

## 可能的产品形态

### MVP版本：电视App
**硬件要求**：智能电视 + USB摄像头 + 手机（家长端）
**核心功能**：
1. **语音唤醒**："小X小X"（类似智能音箱）
2. **互动内容库**：
   - 故事时间（选择分支剧情）
   - 儿歌跟唱（摄像头识别跟跳/跟唱）
   - "这是什么"（摄像头识别孩子手中的物品）
   - 简单游戏（"指一下红色的东西"）
3. **情绪识别**：摄像头检测孩子表情，调整内容节奏
4. **家长端**：
   - 查看"今日互动记录"
   - 远程发起语音（"妈妈很快回来"）
   - 设置"陪伴时长上限"

### 进阶版本：盒子方案
**硬件**：LTU专用盒子（带摄像头麦克风），接任意电视HDMI
**优势**：
- 算力更强（本地LLM）
- 摄像头角度优化（儿童身高视角）
- 隐私设计更彻底（物理遮蔽键）

## 技术栈初步思考

| 模块 | 技术选项 | 考量 |
|------|---------|------|
| **语音交互** | Whisper ASR + 自研TTS | 需要童声优化 |
| **视觉理解** | 轻量级CV模型（YOLO/RTMPose）| 本地运行，识别物品/动作/表情 |
| **对话大脑** | 云端LLM API + 本地缓存 | 成本vs延迟权衡 |
| **内容生成** | RAG + 预设模板 | 安全可控，非纯生成 |
| **多端同步** | WebRTC / 私有协议 | 家长端实时查看 |

## 下一步待验证

1. **技术可行性**：电视端WebRTC/原生App性能测试
2. **用户验证**：找3-5个有3岁孩子的家庭访谈
3. **竞品对标**：国内OTT儿童教育App现状
4. **成本估算**：云端LLM调用成本（按使用时长）
